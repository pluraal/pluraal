The fundamental idea of remarkable is to create tools around Markdown that make it feasible as the only interface that human participants in the SDLC interact with. Spec Driven Development puts Markdown documents in the center of software development and automate the coding using AI agents, which means that more down documents become the primary medium for defining system behavior. While this provides a lot of freedom, it also makes very difficult to follow for human participants, given the unstructured and lengthy nature of these documents. While L alms can breeze through those documents in seconds and build up a knowledge graph, humans don't perform well on the same. Unfortunately, even summarization doesn't help because these documents contain crucial details which need to be reviewed specifically.

This is a difficult problem, but it's very similar to the problem of managing large code bases which we have mastered over decades. The solution is proper structuring, modularization, specialized syntax for certain problem domains, verification tools, debuggers, and many more. None of those is currently available for Mardaan. Markdown as a programming language may sound like a strange idea, but if we step back a little bit and think about how our existing programming languages work, it may sound a bit more realistic. Every programming language starts with a text file. Without the language tooling, a source code file is just that, a text file. Then comes the syntax and the tooling built on top of that. Syntax highlighting, structured views, collapsing sections. These all build on a language parser. Then comes the more sophisticated bits, such as referencing definitions from other files, imports or even package management. Then there's semantics, name resolution, type checking, and finally execution and debugging. All of that starts out with a text file, just like Markdown. In fact, Markdown is a little ahead of the curve, since it already defines some structure. You have hierarchical headings, lists, ordered and unordered tables, references, local and external links, anchors, and even footnotes. That are strangely similar to imports. So if we are we were considering the idea of using Markdown as a programming language, we would be in a much better spot than if we started out with just text.

But the great advantage of Markdown is the freedom it provides for how things are structured, and that unlike programming languages, it is not limited in terms of what you can express. Programming language syntax creates a closed world where tools only allow syntax that is directly supported by the language. For remarkable, we are envisioning an open world approach where unrecognized syntax is just markdown that can still be interpreted by humans and AI agents, but the tooling simply treats it as unknown structure. This has many benefits, one of which is gradual migration, but it also allows language features to be added much easier since each feature is independent. The challenge then becomes how tools would recognize these features, and the answer is through references. It would be incredibly fragile to detect certain syntax patterns in Markdown, and it would create an unworkable environment where if you have a syntax error, instead of reporting an error, the tool would simply skip that section. Imagine how many times people would ask why doesn't it pick it up? This is crucial for the the open world approach. Work and it is the main reason why current programming languages use a closed world approach. However, limiting that is. And here comes a major innovation. Agents are very good at recognizing patterns, especially if detailed instructions are given, so we can use them to enrich the markdown with. Hard references to the specific language features each section encodes. The AI agent can also restructure the section to align with the syntax. Wired This second task can be done with tooling support as well, since as soon as a section is referencing a specific feature, the tooling can highlight syntax issues. This is an invaluable tool for the AI agent that can help pattern recognition as well, especially to filter out false positives in the workflow. The age AI agent would recognize a pattern. Restructure the syntax to align. Called the tool to verify it, and if after repeated tries it couldn't structure it correctly, it could deduce that the pattern doesn't actually align with the feature. Going back to the vision, the idea is that these enriched semi structured markdown documents could be displayed in a rich web view that makes them easier to understand and verify for humans. You can think of that is the as the IDE part of remarkable. There will also be tools that do verification on their own and present the results to the human, such as test runners. Type checkers. When the spec is all verified, we should have enough information to generate the code that executes in production. And this is where another crucial innovation comes in. Once the executable calculation is in production, it should be possible to bind our original spec to it and debug the execution through the reach view that we used to build the original spec. With that, the loop is closed. We have an end to end solution that uses AI agents to raise the abstraction level while still remaining full retaining fuel, full human understanding, better understanding than ever before, while also allowing easy extensions and tailoring to specific domains.
